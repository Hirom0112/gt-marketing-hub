# Live-demo loops

## Stripe webhook idempotency (A3)

Proves the A3 headline against the **local Supabase stack**: a
`checkout.session.completed` event is signature-verified (real Stripe `t=,v1=`
HMAC), recorded once in the `payment` + `stripe_events` ledger, and a **resend of
the same `event.id` dedupes to NOOP** — no double-fulfill.

### Prerequisites
- Local Supabase up (`supabase start`) and `backend/.env.local-supabase` present
  (the provisioning step writes it).
- `stripe` CLI installed (`brew install stripe/stripe-cli/stripe`).
- **You do this once, interactively:** `stripe login`.

### Run
```bash
stripe login                          # one-time, opens a browser
bash scripts/demo/stripe_webhook_loop.sh
```

The script obtains the webhook secret from `stripe listen --print-secret`, starts
the backend on `:8000` (`STRIPE_MODE=simulate`, payments persisted to the local
Supabase), forwards events to `/payments/webhook`, fires one event, then resends
the **same** event id and prints the `stripe_events / payment` counts before and
after — they are unchanged across the resend (the idempotency proof).

### Notes
- Receive/verify/dedupe needs **only** the webhook secret (auto-generated by
  `stripe listen`); no `sk_test_` key. Flip `STRIPE_MODE=live` + set
  `STRIPE_SECRET_KEY=sk_test_…` only when you also want **outbound** Stripe calls.
- A stock `stripe trigger` event has no matching family, so the recorded payment
  row has `family_id = NULL` (the webhook never fabricates a mapping — INV-2/INV-10).
  That does not affect the idempotency proof, which is about the dedupe on
  `event.id`.
